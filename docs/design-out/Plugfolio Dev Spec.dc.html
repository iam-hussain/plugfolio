Product & User-Journey Specification

Every role, every flow, every screen, every state — and how each maps to shadcn/ui + Tailwind and the Plugfolio design tokens. This is the build reference for v1.

Brand source of truth → Plugfolio Brand Guidelines v1.1  ·  Interactive prototype → Plugfolio UI

00 · Orientation

Two rules that shape every screen

Golden rule 1
An account is never the price of shopping.
No modal, wall, or "sign up to continue" may appear on the path from arriving at a creator page to landing on the retailer. Sign-in exists only for "act as yourself" actions (follow, comment).
Golden rule 2
v1 handles no money.
Buy buttons forward to the retailer through the creator's own affiliate link. Never build a cart, checkout, wallet, or payout. Collab terms "settle off-platform."

What Plugfolio is, in one line: a creator turns their content into a page where every post is shoppable; a follower taps a post and buys — no account, no friction. Three roles share one product; one email account can hold any mix of them.

Lime rule — enforce in code review
Electric Lime is a fill only, always with dark text on top. Never lime type on white, never lime as --color-accent text color. It fails AA as a text color on light. Use it for the coupon code chip, the "tracked" flourish, and dark-surface prongs.

Wiring into Tailwind + shadcn/ui

shadcn expects tokens as CSS variables it references through Tailwind's theme. Define the variables in globals.css, map them in tailwind.config, and let dark mode flip via a .dark class on <html>.

/* globals.css */
:root {
  --primary: 262 83% 58%;        /* #7C3AED */
  --primary-foreground: 0 0% 100%;
  --accent: 76 100% 62%;         /* #C6FF3D — dark text only */
  --accent-foreground: 258 26% 9%;
  --background: 252 33% 97%;     /* Canvas */
  --card: 0 0% 100%;
  --foreground: 258 26% 9%;      /* Ink */
  --muted-foreground: 257 9% 34%;
  --border: 258 26% 92%;
  --ring: 258 90% 76%;
  --radius: 0.75rem;
}
.dark {
  --primary: 258 90% 66%;
  --background: 258 26% 9%;
  --card: 256 24% 12%;
  --foreground: 260 40% 96%;
  --border: 256 24% 21%;
  /* accent stays lime; keep --accent-foreground dark */
}

Then in components use only utility classes bound to tokens: bg-primary text-primary-foreground, bg-accent text-accent-foreground, border-border, ring-ring. No hex in JSX, ever.

04 · Journeys

The five user journeys, end to end

Each journey lists where it begins, the step-by-step path (with the UI element the user touches), the success state, and the failure/empty handling. Steps are numbered so QA can script them.

4.1  Shopper — tap → see → buy the product's soul

Begins: a bio-link tap in Instagram/TikTok, inside the app's in-app browser. Target: bio tap → retailer in 3 taps. Zero interruptions, no account.

STEP 1
Bio link tap
Lands on /[handle] creator page.
STEP 2
Tap a post tile
3-col grid; whole tile tappable → post view.
STEP 3
See product
Tagged product card sits right under the media.
STEP 4 · WIN
Tap Buy → retailer
Lime CTA, busy "Opening…", forwards.

Buy behaviour (component-level): tap → button disabled, label → "Opening…" → tap recorded in background → browser navigates to the affiliate URL. It forwards even if recording fails — failure is invisible to the shopper. In-app browsers double-fire taps, so the busy state must make single-action feedback obvious; the backend absorbs duplicates. No interstitial, no confirmation, no account prompt ever.

Success: retailer site opens in ≤3 taps. Empty: "No products tagged on this post." Not found: foreign post/product id → 404.

4.2  Shopper — the one optional account

Begins: tapping Follow, or trying to comment, on a creator page. This is the only account trigger — never fired by a Buy action.

STEP 1
Tap Follow / comment
Outline Follow, or the "Sign in to comment" band.
STEP 2
Sign-in door
Email field → magic link. Reassures: buying needs no account.
STEP 3
Magic link → back
Returns to the same page, now following.
STEP 4 · PAYOFF
/following
Simple list of followed @handle links.

Success: Follow toggles to a quiet "Following"; comment composer appears. Empty /following: "You aren't following anyone yet. Tap Follow on a creator's page." Guard: signed-out visit to /following → sign-in (no flash of protected content).

4.3  Creator — sign up → connect → profile → tag → live

Begins: Landing "Creator" router, or a direct sign-in. The emotional peak is step 5 — "my reel is now shoppable." Design its before/after to feel like a win.

1 · SIGN IN
Email → magic link. Account exists in one step, nothing else asked.
2 · CONNECT
Connect Google (YouTube) / Meta (IG). [LATER] — show explanatory notice until OAuth is live.
3 · PROFILE
"New profile" → instant page with a random handle (creator-a1b2c3d4).
4 · POST
Posts tab → add post (media URL + caption; [LATER] auto-import).
5 · TAG → LIVE
Paste product URL (grabs title/image/price) + affiliate link → Tag product. Post is shoppable.

Success: the tagged product appears in the editor list — the celebratory moment. Silent fallback: an unreadable product page falls back to its hostname as the title (never an error). Errors (inline): no connection (403), "An account holds at most 5 profiles."

4.4  Creator Admin ↔ Manager

Begins: Admin in Settings → Managers. Admin invites a Manager by email (cap 3). The Manager signs in with that email, sees the managed profile (badged "· manager"), and can post, tag, fix products, and work Collabs — but cannot see Settings (the link is hidden and the route is blocked; navigating there silently redirects to /dashboard, leaking nothing). Admin can remove a Manager anytime.

Errors (inline): "A profile has at most 3 Managers", self-invite, invalid email. Empty: "No Managers yet."

4.5  Business ↔ creator collab (two doors)

Door 1 · open board
Business posts a requirement (need, brief, budget text, deadline) → it lists on the open board inside creators' Collabs tab → a creator taps Approach with an opener → thread.
Door 2 · direct
Business browses a creator page → Request collab strip under Follow → thread lands in the creator's Collabs.

Thread → agreement: both sides message; each taps Accept terms (accent). After you accept, your button becomes a quiet "You accepted" (irreversible in v1). When both have accepted, the thread header appends the settled badge: "Agreed — payment settles off-platform."

Success state — the "Agreed" moment
Render as a distinct settled state (badge/banner). The honesty label "payment settles off-platform" must stay — v1 never touches money. Non-participants get a 404: the thread must not exist for them.
05 · Product card

The product card — three faces

Every product card is outbound — never a cart, checkout, or payment on Plugfolio. There are two card kinds plus a coupon that rides on either one. Consider all three together whenever you touch a card.

Face 1 · Affiliate (baseline)
Someone else's product
Photo · title · price · one Buy → retailer via the creator's affiliate link. Creator earns a network commission. No label — commission is invisible to the shopper. This is the default look everything else diverges from.
Face 2 · Creator's own
Their own product
Same layout, two changes only: a quiet "Their own product" trust tag, and the button reads "Shop their store" (not "Buy"). No commission language anywhere on this card.

Face 3 · Coupon — an offer that rides on Face 1 or 2 (not a third type)

A coupon adds a copyable code chip: tap → copies → chip swaps to "Copied ✓" inline (no toast; the code never truncates). Optional expiry line ("Valid till 3 Aug"). After expiry it collapses to a muted "Offer ended" and the card returns to its normal face — the product never disappears. The chip is a sanctioned accent-color moment (the deal is the most persuasive thing on the card) but must not overpower the primary button; mind AA in light and dark.

Three redemption channels — chosen by which fields the creator filled
Channel Fields filled Actions on card
Onlinecode + linkCopy code, then Buy / Shop their store (copy first, go second)
In-storecode + shop note, no linkNo Buy button. Note + code IS the action ("Show this at the counter — Indiranagar store"). Design so the card doesn't look broken.
Bothcode + link + noteAll actions present

Authoring, reporting & the coupon copy state

  • Tagging editor: a kind toggle (Affiliate / My own product) + a collapsed "+ Add a coupon" section (code, expiry, in-store note). The default affiliate flow stays exactly as fast as today.
  • Products tab: rows show the kind tag + a compact coupon summary.
  • Earnings: shows code copies beside taps; in-store copies are labeled "redemption not tracked."
Do NOT build into product cards
Checkout / cart / payments · coupon auto-apply · QR or tracked in-store redemption · bundles / drops · multiple coupons on one product. One coupon per product, copy-only.

07 · States, tone & scope

Cross-cutting rules

Honest states — design all four, none are afterthoughts

Empty
Specific, human line + the next action. Every list has one (see each screen above).
Busy
First tap → disabled + label change ("Opening…"/"Tagging…"/"Saving…"). In-app browsers double-fire; backend absorbs duplicates.
Error
Inline line below the field, role="alert". Specific & human ("A profile has at most 3 Managers") — never a code.
Loading
Skeletons that mirror the real layout; prefers-reduced-motion respected.

Copy & tone

Direct, confident, creator-native; short lines, no hype or jargon. The personality is removing friction. Say the quiet parts loudly and keep these honesty labels verbatim in meaning: "shopping never needs an account," "tracked," "payment settles off-platform." Wording is polishable; the stated meaning is fixed.

Explicitly OUT of v1

Referral rewards · wishlists / price alerts · aggregated feed · ratings / badges · media kits · on-platform payments / payouts · TikTok · AI tag suggestions · >5 profiles / >3 managers / finer roles · vanity usernames · creator-to-creator collabs. Coupons are IN (see §05); still out: coupon auto-apply, QR / tracked in-store redemption, bundles / drops, multiple coupons per product.
